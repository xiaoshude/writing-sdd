# 示例：别再每次都「教育」AI 了

这是一个完整的写作流程示例，展示了如何使用 Writing SDD 系统从零创作一篇技术博客文章。

## 文章信息

| 项目 | 内容 |
|------|------|
| **标题** | 别再每次都「教育」AI 了 |
| **副标题** | 用 SDD 思想驯服 AI，让你的编程原则被一致执行 |
| **平台** | 技术博客 / 公众号 |
| **字数** | ~2500 字 |
| **写作模式** | lessons-learned + thoughts-on-trends |

## 核心论点

> 大多数人用 AI 编程的痛苦，不是 AI 不够强，而是你的意图没有被结构化地表达和积累。

## 文件说明

| 文件 | 说明 |
|------|------|
| `spec.md` | 选题规格 - 文章的元信息和规划 |
| `润色版.md` | 最终版本 - 经过审查和润色的正文 |
| `review.md` | 审查记录 - 5 个 Gate 的检查结果 |

## 流程回顾

```
1. /writing.new "SDD与Prompt工程"
   ↓
2. /writing.spec  → 完善核心定义、解决待澄清问题
   ↓
3. /writing.plan  → 生成 6 段式大纲
   ↓
4. /writing.draft → 生成初稿（~2500字）
   ↓
5. /writing.review → 5 个 Gate 审查
   ↓
6. /writing.polish → 润色输出
   ↓
7. /writing.publish → 标记发布
```

## 关键经验

### 1. spec.md 是关键
- 在动笔前花时间填写核心定义
- 明确「一句话主题」和「核心论点」
- 用 `[NEEDS CLARIFICATION]` 标记不确定的地方

### 2. 写作模式选择
- 这篇文章选择了 `lessons-learned` 模式
- 结构：痛点 → 问题本质 → 解法 → 实操 → 收尾
- 辅以 `thoughts-on-trends` 的行业视角

### 3. 去 AI 味审查
- 审查发现「问题的本质」需要改为「问题出在哪」
- 排比结构需要打散
- 技术词需要口语化

## 学习建议

1. **先读 spec.md**：了解文章的规划和思考过程
2. **对比初稿和润色版**：看看润色改了什么
3. **查看 review.md**：了解审查是如何发现问题的
4. **尝试复制这个流程**：用自己的主题走一遍
