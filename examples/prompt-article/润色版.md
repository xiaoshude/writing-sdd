# 别再每次都「教育」AI 了

**——用 SDD 思想驯服 AI，让你的编程原则被一致执行**

---

你有没有遇到过这种情况？

你让 AI 帮你写一个状态管理的功能，结果它给你整了一堆 `let` 和直接修改对象的代码。你说：「我要数据不可变」。它改了。下一个任务，又忘了。

你在 Cursor 的 global rules 里写了三条原则：
- 数据不可变
- 单向数据流  
- 唯一数据源

但你发现，AI 好像从来没真正「记住」过这些。每次新任务，你都要重新「教育」它一遍。

更崩溃的是——你换个 AI 工具（从 Cursor 到 Claude，从 Copilot 到 ChatGPT），这些原则又得从头说一遍。

**这不是 AI 不够聪明，是你的意图没有被结构化地表达和积累。**

---

## 问题出在哪：碎片化的 prompt，碎片化的执行

我们先想一个问题：为什么你的编程原则写在 rules 里，AI 却不遵守？

原因很简单——**你给的是「碎片化的提示」，不是「结构化的规范」**。

「数据不可变」这四个字，对你来说背后有一整套思维模型：
- 状态变更通过返回新对象
- 不直接修改输入参数
- 使用 spread operator 或 immutable 库

但对 AI 来说，这就是四个字。它没有上下文，不知道在什么场景下用、用到什么程度、遇到冲突怎么取舍。

所以 AI 的行为就像一个记忆力很差的新同事：你说一次，它做一次；下次任务，又忘了。

**prompt 是意图的表达。碎片化的 prompt，只能产生碎片化的执行。**

---

## 一个思路：借鉴 SDD 的核心思想

最近在研究 GitHub 开源的 SDD（Spec-Driven Development，规范驱动开发）方法论，发现了一个有趣的视角。

SDD 的核心理念是：**规范优先，代码服务于规范**。

传统开发里，我们写 PRD、写设计文档，然后开始写代码。文档写完就扔一边，代码才是「真正的成果」。时间一长，文档和代码就脱节了。

SDD 反过来：规范是主要产物，代码是规范的「表达」。你要改功能？先改规范，再重新生成代码。

这个思路迁移到 prompt 工程，就是：

> **prompt 规范优先，单次对话服务于规范。**

什么意思？

不要每次对话都从零开始「教育」AI。而是先构建一套结构化的 prompt 规范，让每次对话都基于这套规范进行。

---

## SDD 中三个值得借鉴的机制

### 1. 模板约束 LLM 输出

SDD 用结构化模板来约束 AI 的输出质量。模板不是简单的「填空题」，而是一套思维框架，强制 AI 按照特定结构思考。

比如 SDD 的功能规范模板要求：
- ✅ 关注用户需要什么（WHAT）以及为什么（WHY）
- ❌ 避免如何实现（HOW）

这种约束让 AI 不会过早跳到实现细节，保持适当的抽象层次。

迁移到 prompt 工程：你的编程原则也应该有模板，不是一句话，而是一套结构：
- 原则名称
- 适用场景
- 具体做法
- 常见违反案例
- 遇到冲突时的优先级

### 2. 标记不确定性

SDD 要求在规范中明确标记不确定的地方，用 `[NEEDS CLARIFICATION]` 标签。

这能防止 AI 做出「看似合理但可能不正确」的假设。与其让 AI 猜测「登录系统用什么认证方式」，不如明确标记需要澄清的地方。

迁移到 prompt 工程：你的 prompt 也可以加这类标记。比如：

> 使用数据不可变原则。当性能要求极高时，可以例外，但需要在注释中说明原因。

### 3. 阶段性关卡验证

SDD 在实现前设置「关卡」，检查是否符合核心原则：
- 简单性关卡：用的组件是否 ≤3 个？
- 反抽象关卡：是否直接使用框架，没有过度封装？

迁移到 prompt 工程：你也可以让 AI 在输出代码前自检：

> 在给出代码前，检查：是否符合数据不可变原则？是否符合单向数据流？如有违反，是否有充分理由？

## 实操：如何构建你的 prompt 体系

说了这么多，具体怎么做？

### 第一步：提炼你的编程原则

别只写「数据不可变」，要展开成结构化的描述：

**编程原则：数据不可变**

**适用场景**
- 所有状态管理相关的代码
- 函数参数处理
- Redux/Zustand 等状态库使用

**具体做法**
- 使用 spread operator 创建新对象
- 数组操作使用 map/filter/reduce，不用 push/splice
- 状态更新返回新对象，不直接修改

**常见违反**
- 直接修改函数参数
- 使用 Array.push()
- 对象属性直接赋值

**优先级**
当与性能需求冲突时，可以在循环内部使用 mutable 操作，但必须在函数边界返回新对象，并在注释中说明原因。

### 第二步：用结构化模板组织

参考一个优秀的 prompt 是怎么组织的。这是 Xuanwo 的 Agent 配置思路（简化版）：

**关于用户与你的角色**
- 假设用户是经验丰富的后端工程师
- 用户重视「Slow is Fast」，关注推理质量和长期可维护性
- 你的目标：在尽量少的往返中给出高质量方案

**编程哲学与质量准则**
- 代码首先是写给人阅读的，机器执行只是副产品
- 优先级：可读性 > 正确性 > 性能 > 代码长度

**工作流：Plan 模式与 Code 模式**
- 复杂任务必须先 Plan 再 Code
- Plan 模式中必须先阅读代码再提建议

看到没？它不是一堆零散的规则，而是分层组织：
1. **角色定义**：你是谁，AI 是谁
2. **核心原则**：不可违背的底线
3. **工作流程**：怎么协作

### 第三步：建立沉淀机制

每次和 AI 协作，如果发现新的 pattern 或踩坑经验，就更新你的 prompt 规范。

比如你发现 AI 总是把错误处理写得很臃肿，就加一条：

> 错误处理：优先使用 early return，避免深层嵌套

这就是「prompt 资产」的沉淀。时间一长，你的 prompt 规范会越来越精准，AI 的输出质量也会稳步提升。

---

## 为什么这样做有效？

因为你把「隐性知识」变成了「显性规范」。

你脑子里的编程原则、最佳实践、踩坑经验，都是隐性知识。你知道，但没有系统地表达出来。

AI 不会读心术。它只能根据你给的 prompt 来工作。

当你把隐性知识结构化、模板化、沉淀下来，你就拥有了一套可复用的「prompt 资产」。换 AI 工具？把规范复制过去就行。新项目？套用同一套模板。

**这不是在教 AI，是在构建系统。**

---

## 少吃低级的苦

说实话，AI 编程里有很多苦是不值得吃的：
- 每次都重复教 AI 同样的原则
- 每个项目都从零开始写 prompt
- 换个工具就要重新调教一遍

这些是「低级的苦」——不会让你成长，只会消耗精力。

把精力留给更高级的挑战吧：
- 思考系统的架构设计
- 探索新的技术方向
- 解决真正有难度的问题

用 SDD 的思想驯服 AI，让规范替你「教育」AI。

你的时间，值得花在更重要的事情上。
